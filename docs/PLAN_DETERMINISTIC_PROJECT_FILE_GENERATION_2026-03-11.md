# Plan - Deterministic Project File Generation

Date: 2026-03-11
Status: proposed
Scope: replace non-deterministic install-time custom-file migration with deterministic generation and persistence rules, while preserving existing project data across reinstall and reinitialization.

## Problem Statement

Current install/reinstall behavior is slow and non-deterministic because several project files are treated as preserved custom Markdown and then migrated file-by-file through Codex.

Observed consequences:

- reinstall duration scales with the number and size of preserved custom files
- migration outcome depends on free-form Codex rewriting instead of a stable renderer
- install, reinstall, and reinitialization can touch files that are actually project data or runtime state
- projects like `gowire` accumulate valid local workflow content that is not clearly separated into:
  - project configuration
  - managed/generated documentation
  - project-owned business data
  - runtime state

This makes the workflow harder to maintain and creates risk during repeated `aidn install` or reinitialization.

## Goals

Introduce a deterministic generation model that:

- removes Codex-based full-file migration from the normal reinstall path
- separates stable project metadata from project business data and runtime state
- makes generated files reproducible from canonical templates plus explicit config inputs
- preserves project-owned data without loss during reinstall or reinitialization
- remains resilient for already-installed repositories such as `gowire`
- keeps install behavior fast, predictable, and idempotent

## Non-Goals

- no attempt to convert all workflow Markdown into database-backed state in this remediation
- no rewrite of runtime cycle/session artifacts in this first pass
- no lossful migration that rewrites project history or baseline content into a reduced schema
- no hidden fallback to LLM/Codex migration when deterministic rendering is available

## Current Findings from `gowire`

The preserved-file comparison shows three different classes of differences.

### Class A - Stable project configuration embedded in Markdown

Primary file:

- `docs/audit/WORKFLOW.md`

Examples:

- `workflow_version`
- `project_name`
- `source_branch`
- runtime profile defaults
- CI capacity policy
- project constraints
- continuity defaults
- snapshot policy

These are durable repository settings and should not require free-form migration.

### Class B - Project-owned business data

Primary files:

- `docs/audit/baseline/current.md`
- `docs/audit/baseline/history.md`
- `docs/audit/parking-lot.md`

Examples:

- current baseline identifier and promotion scope
- baseline history entries and promoted cycles
- project-specific ideas and parked roadmap items
- legacy references and residual risks

This content is not template customization. It is repository data and must survive reinstall untouched.

### Class C - Runtime or operational state

Primary file:

- `docs/audit/snapshots/context-snapshot.md`

Examples:

- current baseline pointer
- open gaps
- working hypotheses
- current next action
- entry point

This is live or semi-live state and should not be regenerated by install.

### Class D - Managed/generated documentation with stale local drift

Primary file:

- `docs/audit/index.md`

Observed issue:

- local content is not valuable custom data; it is mainly an outdated variant of the current managed template

Conclusion:

- `index.md` should become fully managed and overwrite-safe, not preserved custom content

## Design Principle

Do not solve this by making Markdown migration smarter.

Instead:

1. classify every install-managed file into a clear ownership class
2. move stable settings into explicit machine-readable config
3. generate managed documents deterministically from:
   - canonical template
   - explicit config
   - optional structured local extensions
4. leave project-owned data and runtime state outside reinstall overwrite paths

This makes reinstall predictable and removes the main reason for Codex-based migration.

## Target Ownership Model

### 1. Managed / Generated

Definition:

- file is fully reproducible from templates and config
- reinstall may overwrite it deterministically
- drift should be repaired by regeneration, not migration

Examples:

- `docs/audit/WORKFLOW.md`
- `docs/audit/WORKFLOW_SUMMARY.md`
- `docs/audit/CODEX_ONLINE.md`
- `docs/audit/index.md`

### 2. Project-Owned Data

Definition:

- file contains repository memory or business content
- install may create it if absent
- reinstall must not overwrite existing content

Examples:

- `docs/audit/baseline/current.md`
- `docs/audit/baseline/history.md`
- `docs/audit/parking-lot.md`

### 3. Runtime State

Definition:

- file reflects current operating context
- maintained by workflow/runtime commands, not by install

Examples:

- `docs/audit/snapshots/context-snapshot.md`
- `docs/audit/CURRENT-STATE.md`
- `docs/audit/RUNTIME-STATE.md`

### 4. Hybrid Extension Inputs

Definition:

- structured project-local inputs consumed by deterministic generators
- not directly user-facing generated outputs

Recommended new location:

- `.aidn/project/workflow.adapter.json`
- optional future fragments under `.aidn/project/fragments/`

## Recommended Source of Truth Layout

### Canonical machine config

Keep `.aidn/config.json` for runtime/install config only:

- `workflow.sourceBranch`
- runtime profile
- index store mode
- install defaults

Do not overload it with long project policy prose.

### New project adapter config

Add a new structured file:

- `.aidn/project/workflow.adapter.json`

This file should contain stable project adapter metadata such as:

- `projectName`
- `workflowVersionOverride` only if ever needed
- `constraints.runtime`
- `constraints.architecture`
- `constraints.delivery`
- `constraints.additional[]`
- `runtimePolicy.preferredStateMode`
- `runtimePolicy.defaultIndexStore`
- `branchPolicy.allowedCycleTypes[]`
- `branchPolicy.sessionNaming`
- `branchPolicy.cycleNaming`
- `continuityPolicy.defaultsByMode`
- `snapshotPolicy.trigger`
- `snapshotPolicy.owner`
- `snapshotPolicy.freshnessRule`
- `parkingLotPolicy.rule`
- `ciPolicy.capacity`

Optional future extension:

- `.aidn/project/fragments/*.md` for rare long-form project sections that cannot be represented cleanly as structured fields

## Addendum - Adapter Config Wizard

Date: 2026-03-11
Status: accepted

Validated decisions:

1. implement a local wizard solution instead of adding a new prompt dependency
2. expose a reusable dedicated command surface, not only install-time prompting
3. `.aidn/project/workflow.adapter.json` must never be deleted or overwritten by install, reinstall, or reinitialization
4. install behavior:
   - if adapter config is missing and a TTY is available, launch the wizard
   - if adapter config is missing and no TTY is available, stop with an explicit action request
   - if adapter config exists, read it and continue without reopening the wizard unless explicitly requested
5. v1 wizard scope:
   - `list`
   - `add`
   - `edit`
   - `remove`
   - `save`
   - `cancel`
6. v1 field scope starts with the fields required for deterministic generation of managed workflow files

### Why a local wizard is preferred

The repository already uses `node:readline/promises` for interactive install flows.

Benefits of staying local:

- no new dependency to maintain or secure
- consistent behavior with current install prompts
- easier control over TTY detection and non-interactive failure modes
- easier future reuse for additional project settings

### Recommended command model

Introduce a reusable command family such as:

- `aidn project config`
- `aidn project config --wizard`
- `aidn project config --list`
- `aidn project config --set <path>=<value>`
- `aidn project config --remove <path>`

The first implementation should prioritize interactive wizard mode plus explicit read/list support.

### Adapter config persistence rule

`.aidn/project/workflow.adapter.json` becomes a durable project-owned input.

Rules:

- create if missing
- never overwrite automatically once present
- update only through:
  - explicit wizard actions
  - explicit CLI set/remove commands
  - dedicated migration command

### Interaction contract

When the file exists, the wizard should provide a stable review/edit loop:

- display current config
- let the operator choose a section
- allow add/edit/remove on supported fields
- require explicit save
- support cancel with no write

### Non-interactive contract

If the adapter config is missing and the environment is non-interactive:

- do not infer or synthesize the full adapter config silently
- fail with an explicit message explaining how to:
  - rerun in TTY mode
  - provide explicit set values
  - provide an adapter file path

## Detailed Generation Plan

Each generated file should have an explicit deterministic contract.

### Generation 1 - `docs/audit/WORKFLOW.md`

Type:

- managed/generated

Inputs:

- `template/docs_audit/PROJECT_WORKFLOW.md`
- `VERSION`
- `.aidn/config.json`
- `.aidn/project/workflow.adapter.json`
- optional future local fragments

Generation model:

- render canonical metadata placeholders
- render structured project constraints from config arrays/fields
- render branch/runtime/snapshot policy blocks from structured config
- keep canonical sections and ordering owned by template

Overwrite policy:

- reinstall always regenerates this file

User editing rule:

- direct edits to generated regions are not persistent
- project-specific durable customization must move to adapter config or approved fragment inputs

Determinism requirement:

- same template + same config + same fragments = byte-stable output except version/date fields explicitly owned by generator

### Generation 2 - `docs/audit/WORKFLOW_SUMMARY.md`

Type:

- managed/generated

Inputs:

- `template/docs_audit/WORKFLOW_SUMMARY.md`
- `.aidn/config.json`
- `.aidn/project/workflow.adapter.json`

Generation model:

- render source branch and runtime mode references from config
- optionally include project-specific short policy hints derived from adapter config
- preserve canonical quick-reload structure from template

Overwrite policy:

- reinstall always regenerates this file

Determinism requirement:

- summary wording must come from template rules and structured config only

### Generation 3 - `docs/audit/CODEX_ONLINE.md`

Type:

- managed/generated

Inputs:

- `template/codex/README_CodexOnline.md`
- `.aidn/config.json`
- `.aidn/project/workflow.adapter.json`

Generation model:

- keep canonical guidance from template
- inject project runtime mode/source branch references only where needed

Overwrite policy:

- reinstall always regenerates this file

Determinism requirement:

- no project-specific prose should live directly in the generated file

### Generation 4 - `docs/audit/index.md`

Type:

- managed/generated

Inputs:

- `template/docs_audit/index.md`
- `.aidn/config.json`
- possibly selected generated/state file presence

Generation model:

- generate the current index from canonical template
- include current baseline pointer only if that pointer is sourced from a structured field or explicit state reference

Overwrite policy:

- reinstall always regenerates this file

Special note:

- remove this file from the preserved custom file set
- local project drift in this file is not worth migrating

### Generation 5 - `docs/audit/baseline/current.md`

Type:

- project-owned data

Inputs:

- created initially from `template/docs_audit/baseline/current.md`

Update owner:

- workflow/baseline promotion commands, not install

Overwrite policy:

- install creates only if missing
- reinstall never overwrites existing content

Determinism requirement:

- none at install time beyond seeded initial template

### Generation 6 - `docs/audit/baseline/history.md`

Type:

- project-owned data

Inputs:

- created initially from `template/docs_audit/baseline/history.md`

Update owner:

- baseline promotion/history commands

Overwrite policy:

- install creates only if missing
- reinstall never overwrites existing content

### Generation 7 - `docs/audit/parking-lot.md`

Type:

- project-owned data

Inputs:

- created initially from `template/docs_audit/parking-lot.md`

Update owner:

- project users and workflow commands that move ideas between parked and converted states

Overwrite policy:

- install creates only if missing
- reinstall never overwrites existing content

### Generation 8 - `docs/audit/snapshots/context-snapshot.md`

Type:

- runtime state

Inputs:

- runtime state projection, not install template regeneration

Update owner:

- runtime/session workflow commands

Overwrite policy:

- install must not rewrite this file on reinstall
- install may create a seed snapshot only on first initialization if absent and if no runtime state exists

## Implementation Strategy

### Phase 1 - Add ownership metadata to install model

Extend install manifests or install services to support file ownership modes:

- `generated`
- `seed-once`
- `runtime-state`
- `merge-managed` where still needed

Expected result:

- install pipeline knows which files may be overwritten and which may only be created once

### Phase 2 - Introduce project adapter config

Add:

- `.aidn/project/workflow.adapter.json`

Define a normalized schema for:

- project metadata
- constraints
- branch policy
- runtime policy
- continuity policy
- snapshot policy
- CI policy

Expected result:

- stable project adapter settings become machine-owned inputs

### Phase 3 - Build deterministic renderers

Add dedicated renderers for:

- `WORKFLOW.md`
- `WORKFLOW_SUMMARY.md`
- `CODEX_ONLINE.md`
- `index.md`

Renderer responsibilities:

- take canonical template plus structured config
- emit final Markdown
- preserve canonical ordering and wording where possible
- avoid free-form Codex migration

Expected result:

- generated outputs no longer depend on LLM rewrite behavior

### Phase 4 - Remove Codex migration from normal reinstall path

Change install behavior:

- if file is `generated`, regenerate directly
- if file is `seed-once`, preserve existing content
- if file is `runtime-state`, skip on reinstall

Codex migration should become:

- explicit fallback tool only for legacy/manual rescue
- not part of default `aidn install`

Expected result:

- reinstall time becomes bounded by deterministic file generation and artifact import only

### Phase 5 - Migrate already-installed repositories

Add a one-shot migration helper that:

1. reads the current project files
2. extracts stable adapter settings into `.aidn/project/workflow.adapter.json`
3. leaves project-owned data files untouched
4. regenerates managed files from config
5. produces a migration report of extracted fields and preserved files

Expected result:

- existing repositories adopt the deterministic model without data loss

## `gowire` Conversion Plan

`gowire` needs a no-loss migration path.

### Step 1 - Extract stable adapter fields

Read current `gowire` files and extract into adapter config:

- project name
- source branch
- runtime profile
- project constraints
- local CI capacity policy
- continuity defaults
- snapshot policy wording that is truly policy, not state

### Step 2 - Preserve project-owned data as-is

Do not rewrite:

- `docs/audit/baseline/current.md`
- `docs/audit/baseline/history.md`
- `docs/audit/parking-lot.md`
- `docs/audit/snapshots/context-snapshot.md`

Only seed metadata references if strictly needed and only when absent.

### Step 3 - Regenerate managed files

Regenerate:

- `docs/audit/WORKFLOW.md`
- `docs/audit/WORKFLOW_SUMMARY.md`
- `docs/audit/CODEX_ONLINE.md`
- `docs/audit/index.md`

### Step 4 - Validate reinstall and reinitialization resilience

Run after migration:

1. `aidn install --target . --pack core`
2. `aidn install --target . --pack core --verify`
3. project reinitialization flow

Pass condition:

- project-owned data files do not change
- generated files are stable across repeated runs
- no Codex migration is needed in the standard path

## Acceptance Criteria

The plan is complete when:

- stable project adapter settings are no longer stored only as hand-edited Markdown
- `WORKFLOW.md`, `WORKFLOW_SUMMARY.md`, `CODEX_ONLINE.md`, and `index.md` are generated deterministically
- baseline/history/parking-lot are preserved on reinstall
- snapshot/runtime-state files are not overwritten by reinstall
- the default install path no longer uses Codex migration for managed files
- `gowire` can be migrated without data loss
- repeated install + verify + reinit runs are idempotent for preserved files
